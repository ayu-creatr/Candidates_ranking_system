## 1. Problem Statement

Given a pool of **100,000 candidates** and a Job Description for a **Senior AI Engineer** role at Redrob AI (Series A, Pune/Noida, India), rank all candidates from most to least suitable and output a CSV with `candidate_id`, `score [0–1]`, and `reasoning`.

The JD explicitly warns:

> *"The right answer is not 'find candidates whose skills section contains the most AI keywords.' That's a trap we've explicitly built into the dataset."*

The scoring system must reason about the **gap between what the JD says and what it means**.

---

## 2. Solution Overview

A **two-stage, empirically-grounded multi-signal ranker**:

**Stage 1 — `precompute.py`** (run once, ~5–10 min on CPU):
- Encodes the JD and all 100K candidate profiles using `all-MiniLM-L6-v2`
- Measures population-level normalization constants (P1/P99 for semantic, P95 for skills)
- Saves everything to `artifacts/` — rank.py never recomputes these

**Stage 2 — `rank.py`** (run once, ~90 sec):
- Loads pre-computed embeddings from disk
- Scores all 100K candidates using 7 base components + 5 behavioral components
- Outputs a sorted CSV submission

**Key architectural choices:**
- Semantic similarity is **P1/P99 percentile-clipped** (not raw cosine) because MiniLM compresses similarities into a narrow band — rescaling is essential
- Skill scores use a **P95 denominator** measured from the population, not a hardcoded constant
- Behavioral signals act as a **gate modifier** (20% weight) — a behaviorally dead candidate can't rank in the top 10 regardless of how good their base score is
- Consistency is a **hard gate** (not a score component) — 99.96% of candidates score 1.0, so it's near-zero as a differentiator but catches 43 real anomalies

## 5. Architecture — How It Works

### 5.1 Scoring Formula

```
FINAL_SCORE = 0.80 × BASE_SCORE + 0.20 × BEHAVIORAL_SCORE

BASE_SCORE = (
    0.22 × semantic_score      ← P1/P99-rescaled cosine similarity
  + 0.15 × skill_score         ← P95-normalized semantic anchor matching
  + 0.20 × production_score    ← Keyword evidence (T1 + T2 − negs), v6 temporal
  + 0.14 × career_depth_score  ← YoE + tenure-weighted company + stability + v6 title velocity
  + 0.08 × title_score         ← Full-coverage soft gradient (no hard zeroes)
  + 0.08 × ownership_score     ← Ownership signals (hits / 5.0)
  + 0.08 × eval_score          ← Evaluation signals (hits / 6.0)
  + 0.03 × retrieval_score     ← Retrieval/IR signals (hits / 6.0)
  + 0.01 × education_score     ← Institution tier + field relevance
  + 0.01 × location_score      ← Additive, not a multiplier
)

BEHAVIORAL_SCORE = (
    0.30 × availability_score  ← last_active_date recency + open_to_work_flag
  + 0.25 × engagement_score    ← recruiter_response_rate + icr + oar (−1 → 0.50 neutral)
  + 0.20 × momentum_score      ← 30-day: saves + apps + views + search appearances
  + 0.15 × github_score        ← github_activity_score (−1 → 0.15 floor)
  + 0.10 × notice_score        ← notice_period_days alignment
)

HARD GATE: consistency_score < 0.70  →  FINAL_SCORE = 0.0
```

### 5.2 Component Functions

#### `compute_semantic_score(idx)` — 22% of base
Measures how well the candidate's text aligns with the JD.

- **Input:** pre-computed embeddings (no model call at runtime)
- **Formula:** `0.35 × sim(JD, headline+summary) + 0.65 × sim(JD, career_text)`
- **Normalization:** P1/P99 percentile clip — MiniLM cosine scores cluster in `~[0.1, 0.55]`; min-max is outlier-sensitive, P1/P99 is robust
- **Career weighted higher** (0.65 vs 0.35) because summaries are often generic/aspirational while career descriptions contain actual work evidence

#### `compute_skill_score(candidate)` — 15% of base
Semantic anchor matching — not keyword counting.

- 7 JD-aligned skill anchors cover: embeddings/retrieval, evaluation (NDCG/MRR), Python quality, LLM fine-tuning, NLP/IR, recommendation systems, scalability
- Each candidate skill is compared against all 7 anchors via cosine similarity
- Skills below `0.35` cosine floor are ignored (catches HTML, Redux, Excel noise)
- Score weighted by: `max_similarity × proficiency × (duration_bonus) × endorsements × assessment_score`
- **Denominator = P95** of population totals — preserves differentiation at the top without clipping the best candidates

#### `compute_production_score(candidate)` — 20% of base
Evidence that the candidate has shipped real systems, not just studied them.

**T1 keywords** (strong, direct production evidence):
```
"production", "deployed", "real users", "a/b test", "index refresh",
"embedding drift", "ndcg", "mrr", "offline benchmark", "recruiter feedback", ...
```

**T2 keywords** (supporting ownership/context signals):
```
"shipped", "launched", "owned", "bm25", "migrated", "hybrid retrieval",
"improved precision", "recruiter engagement", "click-through rate", ...
```

**Negative signals** (pure research / framework-enthusiast flags):
```
"arxiv", "research lab", "prototype only", "langchain tutorial",
"demo project", "how i built", ...
```

- Score: `0.65 × T1_normalized + 0.35 × T2_normalized − neg_penalty + github_bonus`
- GitHub bonus: +0.10 if gh≥60, +0.05 if gh≥30, 0 if gh<0
- **v6 temporal penalty:** −0.05 if candidate has AI/ML skills but ALL roles started post-2022 (JD: *"people who understood retrieval before it became fashionable"*)

#### `compute_career_depth_score(candidate)` — 14% of base
Four sub-signals combined:

| Sub-signal | Weight | Details |
|---|---|---|
| YoE band | 40% | 4–9yr → 1.0; 10yr → 0.90; 3yr → 0.80; scales down from there |
| Company quality | 35% | Tenure-weighted average across all career roles |
| Stability | 20% | Average tenure per role; ≥36 months → 1.0, <18 months → penalized |
| Industry recency | 5% | Bonus if last 2 roles are in AI/ML, Fintech, E-commerce, SaaS, etc. |

Company scores: Swiggy/Razorpay/CRED = 0.95, TCS/Infosys/Wipro = 0.50 (soft, tenure-weighted — not a hard gate), unknown = 0.65 neutral.

**v6 Title velocity penalty:** if avg tenure < 20 months AND 3+ distinct title levels across career → −0.05 applied to career score. Catches the "title-chaser" anti-pattern the JD explicitly names.

#### `compute_title_score(candidate)` — 8% of base
Soft gradient lookup — no hard zeroes. Full dataset coverage.

| Title | Score | Rationale |
|---|---|---|
| Search Engineer | 1.00 | Exact role match |
| Recommendation Systems Engineer | 1.00 | Exact role match |
| Applied ML / ML Engineer | 0.97–0.98 | Direct fit |
| NLP Engineer | 0.95 | Core domain |
| Computer Vision Engineer | 0.80 | Domain mismatch but still technical AI |
| Senior Software Engineer | 0.78 | Many true AI engineers carry this title |
| Backend Engineer | 0.72 | Common for search infra engineers |
| Software Engineer | 0.68 | Broad, but relevant |
| Business Analyst | 0.22 | Non-technical |
| HR Manager | 0.10 | Irrelevant |
| Unknown titles | 0.35 | Neutral default |

#### `compute_ownership_score(candidate)` — 8% of base
Computes ownership score scaled by hits / 5.0.

- **Signals:** "owned", "owner", "end to end", "e2e", "architected", "designed and built", "built from scratch", "responsible for", "led implementation", "technical lead", "drove migration"
- Hits are scaled by 5.0 and clipped to 1.0.

#### `compute_eval_score(candidate)` — 8% of base
Computes evaluation score scaled by hits / 6.0.

- **Signals:** "ndcg", "mrr", "map", "offline benchmark", "online experiment", "ab test", "a/b test", "evaluation framework", "ranking quality", "precision at k", "recall at k", "recruiter feedback"
- Hits are scaled by 6.0 and clipped to 1.0.

#### `compute_retrieval_score(candidate)` — 3% of base
Computes retrieval score scaled by hits / 6.0.

- **Signals:** "bm25", "dense retrieval", "hybrid retrieval", "learning-to-rank", "learning to rank", "reranking", "re-ranking", "faiss", "hnsw", "ann search", "embedding model", "retrieval quality"
- Hits are scaled by 6.0 and clipped to 1.0.

#### `compute_education_score(candidate)` — 1% of base
Tier + field relevance, capped at 1.0.

#### `compute_location_score(candidate)` — 1% of base
Additive location component using profile.country directly.


#### `compute_behavioral_score(candidate)` — 20% of final
Acts as an availability gate modifier, not a ranking booster.

- **Availability (30%):** last_active_date recency (≤14d→1.0, >180d→0.10) + open_to_work flag
- **Engagement (25%):** recruiter response rate + interview completion + offer acceptance (oar=−1 → 0.50 neutral, not 0.0)
- **Momentum (20%):** 30-day platform activity: saves, applications, views, search appearances
- **GitHub (15%):** github_activity_score (−1 → 0.15 floor); absence for senior engineers is a mild negative
- **Notice period (10%):** ≤30d → 1.0; 31–60d → 0.75; 61–90d → 0.50; >90d → 0.30/0.15

## 6. Design Decisions — Every Weight Has a Number Behind It

All weights were chosen after running EDA on the full 100K dataset:

```
Real competition pool (≥4 AI skills OR AI title):   20,041
In target YoE band (4–9 years):                     10,912
Behaviorally available (open + active <90d):          2,017
India-based + available:                              1,575
Good GitHub score (≥30):                                653  ← true top-tier pool
```

**Why behavioral is 20% (not higher):**
The real contest for top-10 is among ~650 candidates, not 100K. Behavioral signals narrow 100K → 2K → 650 before base scoring differentiates within that pool. A 20% behavioral weight is enough to push inactive candidates (availability ≈ 0.10) out of the top-10 even with strong base scores.

**Why semantic is 22% (highest single weight):**
It's the only signal that can catch a strong candidate who doesn't use the "right keywords" but whose career text is substantively about retrieval/ranking systems. It compensates for the keyword-trap the JD explicitly warns about.

**Why consistency is a gate and not a score component:**
99.96% of candidates score exactly 1.0. Adding it as a weighted component would consume a weight percentage while contributing near-zero differentiation. It works correctly as a hard gate at 0.70 — it catches the 43 genuinely anomalous profiles.

**Why oar=−1 maps to 0.50 (neutral):**
60% of candidates have never received an offer. Treating them as oar=0 (negative) systematically penalizes the majority for having no offer history, which is irrelevant to their fit. Verified safe: top-20 overlap = 18/20 before and after this fix.

**Why location is additive (1%) not a gate:**
75% of candidates are already India-based. A Singapore candidate at 0.97 overall capability cannot be zeroed by geography when the JD says "case-by-case" for outside India.

## 7. Files Reference

### `precompute.py`
Runs once. Builds the embedding index and measures normalization constants.

**Key constants it measures and saves:**
- `semantic_p1`, `semantic_p99` — for P1/P99 rescaling in `compute_semantic_score`
- `skill_denom` — P95 of population skill totals, used as denominator in `compute_skill_score`
- `production_p99` — checked; if < 0.50, `PROD_RESCALE=True` activates automatic rescaling
- `reference_date` — date used for behavioral signal staleness calculations

## 9. Anti-Patterns the System Handles

The JD explicitly names several anti-patterns. Here's how each is handled:

### ❌ "Keyword stuffers" — candidates with all AI keywords but wrong title/background
- **How caught:** Semantic score requires career-text alignment, not just skill matches. A "Marketing Manager" with AI skills listed will score low on `semantic_score` (career text is about campaigns, not retrieval).

### ❌ "Title-chasers" — Senior→Staff→Principal by switching companies every 1.5 years
- **How caught:** `_compute_title_velocity_penalty()` — avg tenure < 20 months AND 3+ distinct title levels → −0.05 to career score.

### ❌ "Framework enthusiasts" — LangChain tutorials, demo projects
- **How caught:** `RESEARCH_NEGS` includes `langchain tutorial`, `demo project`, `how i built` — each adds a −0.05 penalty to production score, capped at −0.20. A candidate with real production work is not hurt by one tutorial post.

### ❌ "Pure services career" — TCS/Infosys/Wipro entire career
- **How caught:** Company score is tenure-weighted. Current Infosys with 4 prior years at Swiggy → company score reflects the Swiggy tenure more heavily. Hard zero only if entire career is services.

### ❌ "Post-2022-only ML experience" — GPT-era framework users without pre-LLM retrieval
- **How caught:** `_has_pre_llm_ai_experience()` — if candidate has AI/ML skills but all their AI/ML roles started in 2022 or later → −0.05 temporal penalty.

### ❌ "Behaviorally unavailable" — Perfect profile, logged in 8 months ago
- **How caught:** availability_score can be as low as 0.10 for >180-day inactive candidates. At 20% behavioral weight, this pushes them below behaviorally active candidates with similar base scores.

### ❌ "Pure researchers" — Academic labs, no production deployment
- **How caught:** `RESEARCH_NEGS` (`arxiv`, `research lab`, `prototype only`) penalizes production score. JD says explicit disqualifier for pure research backgrounds.

## 11. Requirements & Environment

```
# requirements.txt
sentence-transformers==2.7.0   # SBERT models + cosine similarity utilities
numpy==1.26.4                  # PINNED — numpy 2.x breaks sklearn/scipy (ABI conflict)
scikit-learn==1.4.2            # Not used in scoring; available for future use
tqdm==4.66.4                   # Progress bars during precompute
```

**Python:** 3.10+

**Hardware:** CPU is sufficient. The `all-MiniLM-L6-v2` model is 22M parameters (~80 MB). Expected times on a modern CPU:
- Model download (first run): ~2 min
- Encoding 100K summaries (batch_size=256, truncated to 500 chars): ~10–15 min
- Encoding 100K career texts (batch_size=256, truncated to 1000 chars): ~10–15 min
- Skill totals computation: ~60 sec
- Full rank run (rank.py): ~90 sec
- **Total precompute**: ~25–35 min

> **Why truncate?** `all-MiniLM-L6-v2` has a hard 256-token limit. Career descriptions concatenated can exceed 5000 chars, causing the tokenizer to waste time on text that gets truncated anyway. Capping at 1000 chars (≈200 tokens) gives the model everything it can use.

**Memory:** ~3–4 GB peak (numpy arrays for 100K × 384 embeddings × 2 matrices ≈ 300 MB each; skill embed map pkl ≈ 50 MB).

## 12. AI Tools Declaration

This submission was built with AI assistance. Specifically:

- **Architecture and scoring formula:** Developed iteratively across 6 plan versions, with each version grounded in EDA numbers from the actual dataset.
- **EDA:** Run independently using Python scripts against the full 100K candidates.jsonl.
- **JD cross-referencing:** The raw job_description.docx was read in full; v6 changes (PROD_T1/T2 expansions, RESEARCH_NEGS, title velocity penalty, temporal signal) were derived directly from JD text.
- **Code:** Written with AI assistance but every function implements an explicit design decision documented in implementation_plan.md.
- **No data leakage:** Normalization constants (P1/P99, P95) are computed from the population at precompute time and saved to a file. rank.py reads them from disk — it never sees the full population during scoring.

The scoring system is deterministic and fully reproducible from the candidates.jsonl file and the code in this repository.
